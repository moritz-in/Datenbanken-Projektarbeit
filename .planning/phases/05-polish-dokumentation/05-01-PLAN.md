    ## Technische Hintergründe

    ### Warum MySQL B-Trees?

    InnoDB speichert B-Tree-Indizes in 16 KB-Seiten. Ein B-Tree der Höhe H kann
    N = (Seitengröße / Schlüsselgröße)^H Einträge speichern. Bei 1000 Produkten
    und einem VARCHAR(200)-Index beträgt H ≈ 2–3, was O(log N) Suchoperationen
    bedeutet.

    **EXPLAIN-Beleg:**
    ```
    [actual EXPLAIN output]
    ```

    | Spalte | Bedeutung | Wert (Beispiel) |
    |--------|-----------|-----------------|
    | type | Zugriffstyp | ref (Index-Nutzung) |
    | key | Genutzter Index | idx_products_name |
    | rows | Geschätzte Zeilen | [value] |
    | Extra | Optimierung | Using index condition |

    ### HNSW-Parameter (Qdrant)

    Die Qdrant-Collection `products` wurde mit folgenden HNSW-Parametern erstellt:
    - **m=16** — Jeder Knoten hat max. 16 Nachbarverbindungen pro Layer. Höhere Werte
      → bessere Recall, aber mehr Speicher und langsamere Index-Aufbauzeit.
    - **ef_construct=128** — Kandidatenliste beim Index-Aufbau. Höhere Werte → präziserer
      Index, aber langsamerer Aufbau. Bei 1000 Produkten ist 128 großzügig (optimale
      Qualität bei akzeptabler Bauzeit).

    ### Semantische Lücke (Semantic Gap)

    SQL LIKE vergleicht Byte-für-Byte. Eine Anfrage nach „hohe Last" findet nur Produkte,
    die exakt diese Zeichen enthalten. Qdrant kodiert die Anfrage als 384-dimensionalen
    Vektor und findet auch Produkte mit ähnlicher Bedeutung (z. B. „Hochlastlager",
    „Heavy-Duty-Bearing"), ohne dass das Wort „hohe Last" vorkommt.
